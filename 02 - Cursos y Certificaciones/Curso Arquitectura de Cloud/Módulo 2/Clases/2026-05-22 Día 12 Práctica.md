# Práctica

# Despliegue de arquitectura con docker y balanceador. Para simular una página con demanda.

### Preparación

1. Necesitamos ciertas carpetas y archivos:
	Una carpeta llamada app que contendrá el Dockerfile (un texto plano con recetas de sentencias) y un index.php (un archivo de programación para paginas web)
	
	Una carpeta llamada nginx que contiene un archivo nginx.conf (es un archivo se configuracion del servicio web, parecido a apache, que esta funcionando como ALB Application Load Balancer que distribuye el tráfico hacia el servicio web-app de Docker)
	
	Un archivo compose.yaml (será el guiador del docker, con dos bloques de código, los servicios del balanceador y los servicios de web-app)

### Despliegue del docker compose para alta demanda

1. Arrancamos el docker:
	Nos hubicamos en la carpeta que contiene todo y tiramos `sudo docker compose up -d --build` 
	![](../../../../04%20-%20Otros/Imagenes/Pasted%20image%2020260607033318.png)
	
	Si ocurriera un error se puede tirar `sudo docker logs "4 numemos de container ID"`
	
	Podemos verlo en el navegador con `localhost:8081`
	![](../../../../04%20-%20Otros/Imagenes/Pasted%20image%2020260607034353.png)

2. Nos preparamos para la alta disponibilidad:
	Tiramos `sudo docker compose up -d --scale web-app=3` que nos creara 2 servidores más como lo hace el ASG Auto Scaling Group 
	![](../../../../04%20-%20Otros/Imagenes/Pasted%20image%2020260607035024.png)
	![](../../../../04%20-%20Otros/Imagenes/Pasted%20image%2020260607035141.png)
	
	Y cada vez que agreguemos servidores, reiniciamos el servicio nvginx con `sudo docker restart class2-balanceador-1`para que empiece a balancear las demandas, que simularemos con curl o recargando el navegador
	![](../../../../04%20-%20Otros/Imagenes/Pasted%20image%2020260607040033.png)
	

### Finalizar servidores para volver al tráfico normal

1. Dejamos solo un servidor como estaba funcionando antes de la alta demanda:
	`sudo docker compose up -d --scale web-app=1` dejamos uno solo ya que los otros los borra. Y reiniciamos nginx `sudo docker restart class2-balanceador-1` 
	![](../../../../04%20-%20Otros/Imagenes/Pasted%20image%2020260607040912.png)
	 Podemos ver que solo contesta el mismo contenedor y los otros ya no existen
	 ![](../../../../04%20-%20Otros/Imagenes/Pasted%20image%2020260607041026.png)
	 

### Terminar práctica

1. Limpiamos:
	con `sudo docker compose down` se apaga y elimina todo.
	![](../../../../04%20-%20Otros/Imagenes/Pasted%20image%2020260607041934.png)
	![](../../../../04%20-%20Otros/Imagenes/Pasted%20image%2020260607042000.png)
	


# Guía del Profesor
  

### Paso 1: El archivo index.php (La App Dinámica)

Para que cambie el texto según el contenedor que responda, usamos una línea simple de PHP que lee el nombre del servidor. Crea una carpeta llamada app y dentro coloca este index.php

### Paso 2: El Dockerfile de la App

Dentro de la misma carpeta app, crea el Dockerfile para empaquetar esta mini-web

### Paso 3: El archivo nginx.conf (El Balanceador)

Crea una carpeta llamada nginx al mismo nivel que app, y dentro coloca la configuración que le dice a Nginx que distribuya el tráfico hacia el servicio web-app de Docker

### Paso 4: El docker-compose.yml (El Orquestador)

En la raíz de tu proyecto (fuera de las carpetas app y nginx), crea el archivo maestro

### Paso 5: El "Show" en Vivo para la Clase (La Dinámica)

Una vez que tengas los archivos listos, ejecuta la demo ante tu alumno siguiendo estos pasos:

  

Arrancar el entorno inicial:

  

Bash

docker compose up -d

Entra a http://localhost:8080 y dale a actualizar (F5) varias veces. El alumno verá que el Contenedor ID es siempre el mismo porque solo hay un servidor corriendo.

El Momento de Magia: ¡Escalar en Caliente!

"Llegó el Black Friday, un servidor no da abasto. Vamos a aplicar Elasticidad Horizontal con un solo comando". Ejecuta en tu terminal:

  

Bash

docker compose up -d --scale web-app=3

La Validación del Balanceo (Round Robin):

Vuelvan al navegador en http://localhost:8080. Pídele al alumno que presione F5 repetidamente. ¡Verán cómo el ID del contenedor cambia en cada clic! Nginx está repartiendo una petición a cada contenedor de forma cíclica.

  

Simular una Caída (Alta Disponibilidad / Fault Tolerance):

Abre otra terminal y "mata" uno de los contenedores de la app a la fuerza usando docker stop <id_de_un_contenedor>. Vuelvan a actualizar el navegador: la web sigue online sin enterarse, porque Nginx detecta que ese nodo no responde y redirige el tráfico a los otros dos vivos.
